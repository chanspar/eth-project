# 🚀 Ethereum ETL 파이프라인 최적화 보고서 (2026-05-12)

오늘의 작업은 **Alchemy 무료 티어(Free Tier)**의 엄격한 제한을 극복하고, **일일 300만 건 규모의 대용량 데이터**를 안정적으로 처리하는 데 집중했습니다.

---

## 1. 🛑 Alchemy 429 (Rate Limit) 에러 해결
무료 티어의 초당 CU(Compute Units) 제한(330 CU/s)을 초과하지 않도록 다각도로 Throttling을 적용했습니다.

### 🐢 수동 청크 분할 및 지연 (Manual Throttling)
- **대상**: `extract_receipts_and_logs_task`
- **조치**: 1,300여 개의 트랜잭션 해시를 **100개씩 쪼개서(Chunking)** 요청하고, 각 요청 사이에 **2초의 휴식(Sleep)**을 부여했습니다.
- **결과**: 초당 CU 소모량을 한계치 이내로 묶어두어 429 에러 없이 완주 성공.

### 🧊 태스크 간 쿨다운(Cool-down) 도입
- **조치**: 대용량 블록 추출(3.4GB) 작업 직후, 다음 작업 시작 전 **60초간 대기** 로직 추가.
- **효과**: Alchemy 서버의 Burst Limit 윈도우가 초기화될 시간을 벌어줌.

## 2. 🛡 파이프라인 안정성 및 복구 로직 강화

### 📦 태스크 병합 (Task Merging)
- **변경**: `token_transfers`와 `contracts` 태스크를 하나로 통합.
- **이유**: 로컬 파일을 긴밀하게 공유하는 두 태스크를 분리할 경우, 재시도 시 파일 유실(`FileNotFoundError`) 가능성이 높음. 합침으로써 관리 포인트 일원화 및 안정성 확보.

### 💾 원본 데이터 보호 (File Preservation)
- **조치**: 다운스트림 태스크(`receipts`)가 실패하더라도 업스트림의 결과물(`transactions.json`)을 지우지 않도록 `finally` 블록 수정.
- **효과**: 태스크 실패 시 3.4GB 데이터를 다시 처음부터 뽑지 않고, 그 지점부터 즉시 재시도 가능.

### 🔄 순차적 실행 (Waterfall Orchestration)
- **기존**: 여러 ETL 태스크가 병렬로 실행되어 API 부하 가중.
- **변경**: `Blocks` → `Receipts` → `Transfers` 순서로 **기차놀이(Sequential)** 방식 적용.
- **효과**: 한 번에 하나의 API 요청만 발생시켜 429 에러 원천 차단.

- 시작 시간, 종료 시간, 소요 시간(Duration)을 직관적으로 리포팅.

- **Retry Delay**: 3분 (제한이 확실히 풀릴 때까지 대기)
